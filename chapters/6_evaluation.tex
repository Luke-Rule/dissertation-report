\chapter{Evaluation}
\label{cha:evaluation}

To increase confidence in both the correctness and performance of the FS implementation, evaluation was conducted through multiple stages of testing and performance analysis. 

\section{Validation}

Extensive validation was conducted to evaluate the correctness, robustness and reliability of the FS under both normal and adverse conditions. Given the range of supported operations, numerous edge cases and error scenarios arise. Additionally, the presence of multiple clients and asynchronous communication introduces further complexity, requiring testing not only to verify functional correctness but also to ensure that the system maintains consistency and stability under interleaved workloads.

\subsection{Unit Tests}

Unit tests were developed for the core of the file system, including components such as block, inode and directory management and file handle operations. These tests focused solely on verifying the correctness of the file server’s data structures and their accesses in isolation.

Testing at this level enabled a modular development approach, allowing components to be validated incrementally during implementation. This also simplified debugging, as errors were easier to localise due to their limited scope.

To ensure separation of concerns, these components were tested by directly invoking their functions outside the Microkit environment. This allowed inputs and supporting data to be simulated through deterministic flows and the internal state of the FS to be inspected, enabling correctness to be validated at each stage of execution. Testing regular inputs alongside failure conditions, such as resource exhaustion, gave confidence that the fundamental building blocks of the FS operated correctly before integration and would not cause wider issues upon failure.

For example, one test constructs a mock `indirect block' containing a list of allocated block indices, invokes the \texttt{release\_indirect\_block} routine and then verifies that each referenced block is marked free in the BAT.

\newpage
\subsection{Integration Tests}

Once the components were verified, integration testing was used to validate the behaviour of the system as a whole, exercising all FS operations through its regular running microkit environment. Rather than directly inspecting internal state, validation was performed through the responses and error codes returned to clients, such as creating a file then listing the root directory. This ensured that testing reflected real system usage and confirmed that correct behaviour was exposed through the public interface.


\subsubsection{Single Client}

These tests exercised the majority of functionality from the client’s perspective, covering the full lifecycle of files and directories without the added complexity of interleaved access. Within this scope, it was possible to verify that invalid client inputs were correctly rejected with appropriate error codes, for example when attempting to delete a non-existent file. Additionally, boundary conditions were tested to ensure correct behaviour under extreme cases, such as reading the maximum supported amount of data.

\subsubsection{Multiple Clients}

To ensure that all clients were serviced correctly, and that the system operated reliably under load, extended tests were conducted which interleaved operations from multiple clients. This enabled validation of scenarios that cannot be observed in a single-client context, including permission enforcement and modification of shared resources.

As clients execute independently, explicit synchronisation was required. This was achieved through the use of marker files, where files where created with predefined names and were monitored by other clients. These acted as barriers, allowing coordination of execution order and enabling clients to signal completion of operations to both the server and one another. For example, one test creates a publicly readable file as client 1, after which client 2 attempts to read that file to verify access is permitted. The file is then modified by client 1 to be private and client 2 subsequently attempts to read it again, verifying that the operation is correctly rejected with a permissions error.

\section{Performance Evaluation}

\subsection{Goals}

The primary goal of this evaluation was to compare the performance characteristics of the synchronous and asynchronous FS designs. In particular, the evaluation aimed to quantify the impact of IPC overhead and assess the effectiveness of batching in reducing operation cost. Additionally, the behaviour of both systems under multiclient workloads was examined to determine how well each design scales under contention.

\subsection{Metrics}

For the per-operation benchmarking, performance was evaluated using the average time per-operation, taken from the total execution time of a long sequence of the specified operation, to amortise any stochastic effects. For the more realistic workflows, time taken to complete the whole suite was used.

Time measurement was performed using the CPU cycle counter, as the low-level execution environment did not provide native timing facilities. The cycle count was recorded before and after each client’s measurable operations. Results were averaged across multiple runs to mitigate run-to-run noise and improve reliability.

In multiclient scenarios, fairness was also considered. This was validated by recording the total time taken by each client to complete its workload.

\subsection{Experimental Setup}

All experiments were conducted within the same emulated environment using QEMU's default \texttt{virt} platf-\newline orm\cite{qemu}, ensuring consistency across all tests. The synchronous and asynchronous implementations were evaluated under identical conditions, with the same workloads, file sizes and FS state.

All operations were validated for correctness during execution to ensure that performance results reflected valid system behaviour rather than erroneous outcomes. During the varied workload, this processing also mimicked non-file I/O that would occur in a typical system, helping measured performance reflect realistic operating conditions.

\subsection{Benchmark Design} 
 
Two benchmark suites were developed to evaluate system performance. The first was designed to isolate and quantify the reduction in communication overhead achieved by the asynchronous server, while the second evaluated overall system performance under more realistic and varied workloads.

This initial aim was achieved through a single-client benchmark which repeatedly executed a single type of operation allowing controlled measurements of individual operation costs for both systems. Both read and write operations were evaluated independently, as they are representative of common FS usage and exhibit differing behaviour. As timing was recorded from the client’s perspective, this included both internal server processing time and communication overhead.
 
For the asynchronous implementation, the benchmark was executed with varying batch sizes. A batch size of one provided a baseline comparison, to ensure that the asynchronous design did not significantly degrade performance when batching was not possible, which is important as many real workloads contain inherent dependencies, while larger batch sizes allowed evaluation of how less IPC reduces per-operation cost.
\newpage
The second benchmark was a multiclient, realistic workload designed to simulate typical FS usage. Multiple clients executed independently, each performing a sequence of varied operations across many files. This sequence represented a complete file lifecycle. Operations were interleaved across clients to introduce contention and better reflect real-world usage patterns.

As batching is central to the performance benefits of the asynchronous design, this benchmark was structured to allow batching where it would naturally occur, without artificially constraining the workload, as a workload with strict dependencies between operations would remove the asynchronous benefits and provide limited insight beyond the single-operation baseline benchmark. 

Operations were grouped as such:

\begin{itemize}[itemsep=0.2ex, topsep=0.2ex]
    \item \textbf{Batch 1:} Create.
    \item \textbf{Batch 2:} Write and Close.
    \item \textbf{Batch 3:} Open.
    \item \textbf{Batch 4:} Seek and Read and Delete.
\end{itemize}

Performance was measured across varying numbers of clients, enabling analysis of scalability and contention to provide a more balanced comparison between the designs.

\subsection{Results and Analysis}

\subsubsection{Single-Client Per-Operation Benchmark}
\vspace{0.4cm}
\begin{figure}[H]
    \centering
    \includegraphics[width=0.7\linewidth]{chapters/images/avg_operation_time_by_batch_size.png}
    \caption{Average ticks per operation by batch size (Read and Write)}
    \label{fig:avg_operation_time_by_batch_size}
\end{figure}

Figure~\ref{fig:avg_operation_time_by_batch_size} shows how the average time taken per operation is affected by batch size for both read and write operations, alongside the synchronous baseline for comparison. The synchronous system achieves approximately 4375 cycles for writes and 3625 for reads.

At a batch size of one, asynchronous performance is effectively identical to the synchronous case, with writes and reads taking around 4313 and 3562 cycles, respectively. This confirms that negligible overhead is introduced in the absence of batching, validating that it can be used without performance degradation in more dependent workflows.

As batch size increases, a substantial reduction in per-operation cost is observed, indicating that communication overhead forms a significant component of total operation latency. For write operations, latency decreases to less than 1800 cycles at batch sizes of seven and above, corresponding to an approximate 60\% reduction. Read operations, with less internal processing cost, exhibit an even greater relative improvement, decreasing to around 1000 cycles, yielding reductions of 70\%.

This aligns with expected results. At a batch size of 7, six IPC round trips are avoided. The total reduction of approximately 2500 cycles therefore corresponds to roughly 420 cycles saved per IPC round trip. This is consistent with the expected cost of IPC in seL4, at approximately 200 cycles per direction\cite{klein_sel4_2009}.

The majority of this improvement occurs at relatively small batch sizes, with both the write and read operations exhibiting extremely similar trends, showing this that behaviour is consistent. 90\% of the total reduction seen was achieved by a batch size of 4, beyond this point, performance improvements become incremental, plateauing completely beyond a batch size of 7, as the system approaches its performance bound. This trend is further supported by Figure \ref{fig:avg_operation_time_gradient_by_batch_size}. The gradient magnitude of the per-operation time for both operations decreases rapidly as batch size increases, approaching zero at larger batch sizes.

The stabilising of performance at larger batch sizes indicated that the communication overhead had been effectively amortised, and the system is now operating close to its limit. At a batch size of 7, this upper bound is relatively achievable, indicating that IPC overhead could be effectively mitigated in realistic system conditions.

\subsubsection{Realistic Multiclient Workload}

\begin{figure}[H]
    \centering

    \caption{Total ticks per client by number of clients (Synchronous vs Asynchronous)}
    \label{fig:total_execution_time_by_clients}
\end{figure}
\vspace{-0.2cm}

Figure \ref{fig:total_execution_time_by_clients} shows the mean execution time per client as a function of the number of concurrent clients for both the synchronous and asynchronous systems. Across all client counts, both systems exhibit relatively stable performance, with the asynchronous design consistently outperforming the synchronous baseline. 

For a single client, the synchronous system took approximately $14500000$ cycles to complete the suite, compared to around $8750000$ for the asynchronous system, representing a reduction of approximately 40\%. This improvement remains consistent as the number of clients increases. Expectedly, this reduction is lower than the single-operation benchmark, as there was less opportunity for batching due to dependencies between operations, and the presence of other non-file I/O processing reduced the proportion of time spent on IPC. However, the still substantial improvements across all client counts confirm that the benefits of reduced communication overhead are maintained at high levels of contentention and in more complex and realistic workloads.  

Although there is some variation across client counts, it is minimal and exhibits no clear trend, suggesting that the workload scales predictably and that the systems remain robust as load increases.
