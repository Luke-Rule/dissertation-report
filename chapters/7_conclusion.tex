\chapter{Conclusion}
\label{cha:conclusion}

This dissertation investigated whether IPC overhead in seL4-based systems could be mitigated through architectural and interface-level design choices, specifically through an asynchronous file server. This required the design and implementation of a complete user-space system, including both the file server and the supporting communication mechanisms needed to realise and evaluate the interface model. The work demonstrates that meaningful performance improvements can be achieved through communication-aware system structure, while also showing that IPC overhead is strongly influenced by higher-level architectural decisions beyond kernel mechanisms alone.

\section{Contributions}

This work contributes a complete and practical user-space file server for the seL4 Microkit framework, supporting multiclient access, a simple FS API, and both synchronous and asynchronous communication interfaces. It demonstrates that complex, stateful services can be realised in user space within seL4 while maintaining structured interaction, correctness and practical usability.

This work also contributes empirical insight into mitigation of IPC overhead through asynchronous communication. Evaluation showed that batching can substantially improve throughput and reduce average operation latency, while also demonstrating that these benefits are strongly workload-dependent.

\section{Drawbacks and Limitations}

Despite achieving its objectives well, the system has several limitations that constrain both its performance and general applicability.

\subsection{FS Limitations}

The implemented FS is functionally rich and reasonably well validated, but could be improved in multiple areas. Additional operations such as file renaming and copying could be introduced to improve usability. Furthermore, lookup performance is suboptimal due to linear search mechanisms; the absence of more efficient indexing structures (e.g. hash tables or trees) contributes to increased latency as the file set grows, yet the current implementation is sufficient for general use and for the experiments undertaken in this report.

\subsection{Benchmarking Constraints}

Evaluation required the development of custom benchmarks due to the specialised environment, low-level platform and non-standard interface. While these benchmarks effectively demonstrate relative performance trends, they limit direct comparability with existing systems. Consequently, results should be interpreted as indicative rather than absolute.

\subsection{Lack of Persistent Storage}

The system operates entirely in memory, with no backing disk. Incorporating persistent storage would introduce substantial additional complexity, particularly around buffering, write-back policies and cache management, beyond the scope of this work. Its absence also results in highly deterministic and relatively low operation latencies. In realistic storage-backed systems, higher and more variable I/O latency would reduce the relative impact of IPC overhead, meaning these results likely represent an upper bound on the benefits of communication overhead reduction in practice.

However, such systems would typically operate on multi-core hardware, where an asynchronous design such as the one presented here could offset some of this reduced relative benefit through increased concurrency and non-blocking execution.

\subsection{Single-Threaded Execution}

The server and clients operate on a single thread, limiting concurrency and reducing the potential benefits of the asynchronous design. As a result, observed performance gains arise primarily from reduced communication overhead through batching rather than true parallel execution.

This likely makes the measured gains a conservative estimate of the improvements achievable in a fully concurrent multicore system, where additional benefits from overlapping request generation, processing and response handling could be realised. The architecture is designed to be transferable to such an environment with only minor modification, once supported within seL4.

\subsection{Additional Client Burden}

The asynchronous design introduces additional client-side complexity through management of request queues, response buffers and batching decisions, increasing both per-client memory usage and implementation burden relative to the synchronous model.

Although a supporting library abstracts much of this functionality, clients must still determine how to group operations into batches and when to synchronise, creating a workload-dependent trade-off between throughput and responsiveness. Thus, while the asynchronous model improves performance, it shifts part of the system complexity from the server to clients and may make correct usage more demanding.

\section{Future Work}

Future work should focus on extending the system to a multicore environment, when available, to fully realise concurrency benefits. Additionally, enhancing the FS to support a broader set of operations and more efficient data structures, such as tree-based indexing, would improve scalability, and introducing a caching layer and persistent storage would enable more realistic deployment scenarios. Finally, further exploration of IPC mechanisms, including hybrid synchronous–-asynchronous models and adaptive batching strategies, could also yield additional performance improvements.
