\chapter{Implementation}
\label{cha:implementation}

This section describes the implementation of the file server, building upon the designs presented in the previous section. It details how both the interface and file system components were realised in practice, including data structures, algorithms, communication mechanisms and API abstractions. 

\section{Development Environment and Tools}

The system was implemented in C using the Microkit framework for seL4, specifically version 2.0.1 of the Microkit SDK\cite{}. The build process was managed using a Make-based system, which coordinated compilation, linking and generation of Microkit system images.

All development and testing were conducted in an emulated environment using QEMU\cite{}, targeting the \texttt{virt} generic virtual machine model. This allowed rapid iteration without requiring physical hardware, though it limited evaluation of hardware-specific performance characteristics.

Debugging and verification were primarily performed using logging via serial output, enabling inspection of system behaviour during execution. Additional support tools included Git\cite{} for version control and reproducibility of development progress.

\section{Microkit Framework Setup}

The system is built using the Microkit framework for seL4, which provides a declarative mechanism for defining execution structure, memory layout and inter-process communication\cite{}. This is specified using an XML-based description, which is used to generate a complete system image comprising multiple protection domains. A single client example of this system definition is shown in Listing~\ref{lst:system}.

Each protection domain represents an independent user-space process with its own address space and execution context. Hence, separate protection domains were defined for the file server and each client. Program images were statically assigned to each domain, and memory regions were explicitly declared and mapped into their address spaces at specified addresses. A dedicated memory region for file system state was declared and mapped solely to the server. Additionally, for each client, regions to hold submission and completion queues and associated buffers were declared and mapped into both the file server and the corresponding client, each with the appropriate permissions, allowing shared access.

Alongside this shared memory, IPC between protection domains was established using channels, as detailed in Section~\ref{sec:microkitbg}, with each client being connected to the file server. By defining memory regions and notification channels through Microkit, capability management of all objects is handled automatically by the framework, significantly reducing the complexity of managing and securing IPC. 
\vspace{0.5cm}
\begin{lstlisting}[caption={Microkit system definition.}, label={lst:system}]
     <!-- File Server Memory Region -->
    <memory_region name="fs_memory" size="0x10133000" />

    <!-- File Server Protection Domain -->
    <protection_domain name="file_server" priority="1" >
        <program_image path="file_server.elf"/>

        <map mr="fs_memory" vaddr="0x30000000" perms="rw" cached="true"
          setvar_vaddr="fs_memory_base"/>
          
        <map mr="client_0" vaddr="0x40133000" perms="rw" cached="true"
          setvar_vaddr="clients_memory_base"/>

    </protection_domain>

     <!-- Client Memory Regions -->
    <memory_region name="client_0" size="0x81000" />

    <!-- Client Protection Domains -->
    <protection_domain name="client_0" priority="0" >
        <program_image path="client0.elf"/>
        <map mr="client_0" vaddr="0x30000000" perms="rw" cached="true"
          setvar_vaddr="fs_data_base"/>
    </protection_domain>

    <!-- Client Communication Channels -->
    <channel>
        <end pd="file_server" id="0"/>
        <end pd="client_0" id="0" pp="true"/>
    </channel>
\end{lstlisting}
\newpage
As Microkit systems are statically defined, the number of clients and their associated resources must be specified at build time. To support varying workloads, a script was developed to generate the system description for a configurable number of clients automatically, ensuring consistency while avoiding manual duplication.

In addition to structural configuration, Microkit also requires specification of priority and scheduling parameters for each protection domain. These priorities determine the order in which domains are scheduled by the seL4 kernel, directly influencing system responsiveness and fairness. The file server was, therefore, assigned a higher priority than the client domains to ensure that incoming requests were serviced promptly, reducing queuing latency and preventing clients from stalling due to delayed processing. Client domains were assigned equal priorities relative to one another to maintain fairness across request sources. This scheduling configuration was particularly important in the asynchronous design, where the server must efficiently respond to notifications and process queued operations without being preempted excessively. Consequently, as seL4 utilises a round-robin scheduling policy\cite{}, careful time budget allocation was required to ensure that IPC handling remained responsive while still allowing clients sufficient execution time to generate and consume requests.

\section{Interface}

\subsection{Synchronous Server Model}

This is the simpler of the two models, and as such, was implemented first to validate the correctness of the file system, before introducing the additional complexity of the asynchronous design. 

\subsubsection{Client-Server Communication}

Microkit’s protected procedure call (\texttt{microkit\_ppcall}) provides both the blocking semantics and message passing we require within a single operation. It transfers control from the calling thread to the callee, suspending the caller until the operation returns. As such, the callee must execute at a higher priority than that of the caller, which aligns with our design as we already have the file server at the highest priority for scheduling preference.

In addition to control transfer, \texttt{microkit\_ppcall} supports the transmission of up to 64 words of data in both the request and response. As file names and data are handled via shared buffers, this capacity is sufficient to encode the operation code and associated parameters or return values. Message size is configurable, allowing each operation to transmit only the required data. Consequently, the server must validate the message size after decoding the operation to ensure all necessary parameters are present. 

\newpage
The client request flow proceeds as follows:
\begin{itemize}
    \item The client writes any required data (such as file names or file contents) into its shared buffer.
    \item A message is constructed containing the operation code and relevant parameters.
    \item The client invokes the file server via \texttt{microkit\_ppcall} on its dedicated channel, passing the constructed message.
    \item The client thread is automatically suspended until the operation completes.
\end{itemize}

When control is passed to the server, execution begins at a designated entry point. This is defined as the function signature:

\vspace{0.5cm}
\begin{lstlisting}
microkit_msginfo protected(microkit_channel channel,
                           microkit_msginfo msginfo
                          )
\end{lstlisting}

Since all clients share this entry point, the \texttt{channel} parameter is used to identify the calling client. These channel identifiers are statically assigned in the Microkit system configuration and cannot be modified at runtime, ensuring that requests are correctly attributed. 

Alongside extracting the operation code and parameters from the message the server may also access the client’s shared buffer. All client buffers are mapped into the server’s address space at specified virtual addresses during system initialisation. By arranging the client buffers sequentially - in the same order as the channels, and hence the client IDs - the server can treat them as an array and index into it directly using the client’s channel identifier to locate the appropriate buffer. 

Once the operation is complete, the server constructs a response message containing the result code and any return values, and returns it from the protected function. This automatically delivers the response to the corresponding client and resumes the client thread, which can then inspect the result and any updated buffer contents. 

\subsubsection{API}

To simplify usage and ensure consistent interaction with the file server, the communication and buffer management logic is encapsulated within a set of wrapper functions. These abstractions hide the details of message construction and extraction, buffer usage, and IPC invocation. 

To reduce the number of parameters required for these library calls, each operation has a specific function that can be called, which accepts only the parameters required for that operation and returns an operation unique struct for maximum access simplicity.

For example, the \texttt{send\_open\_file\_request} function takes the desired access mode, a pointer to the file name, and the address of the client’s shared buffer. It copies the file name into the buffer, constructs a new message containing the \texttt{OPEN} operation code and mode parameter, and invokes the file server. Upon completion, it forms a \texttt{result\_code\_file\_id} struct with the result code from the operation and the file ID returned. This struct is then returned to the client, allowing the outcome to be easily interpreted.

\subsection{Asynchronous Server Model}

This is the more complex of the two models and was developed after the synchronous implementation to reduce IPC overhead through asynchronous, queue-based communication. 

\subsubsection{Client-Server Communication}
Rather than transferring control directly to the file server for each operation, the asynchronous model implements communication through shared memory and notification-based signalling. Clients submit requests by writing to shared data structures and continue execution until they explicitly check for completion. 

Each client is assigned a submission queue, a completion queue, and corresponding buffer pools for transferring file-related data. Submission buffers are used to store request data such as file paths or write contents, while completion buffers hold data returned by the server. Allocation tables are maintained for both buffer pools to track usage.  A full operation flow example is displayed in Figure \ref{fig:queues}.
\vspace{0.4cm}
\begin{figure}[H]
    \centering
    \includegraphics[width=0.72\linewidth]{chapters/images/queues.png}
    \caption{Asynchronous client-server communication via shared queues and buffers.}
    \label{fig:queues}
\end{figure}

For operations requiring data transfer, the client first acquires a submission buffer by locating a free entry in the allocation table, as shown in Figure~\ref{fig:buffers}. If no buffer is available, the client must wait until its previously issued operations complete and their resources are released. 
 
\begin{figure}[H]
    \centering
    \includegraphics[width=0.4\linewidth]{chapters/images/buffers.png}
    \caption{Buffer management in asynchronous communication.}
    \label{fig:buffers}
\end{figure}

Once a buffer is acquired, the client writes the required data and constructs a submission queue entry describing the requested operation, including its parameters and the associated buffer index. The structure of this entry is shown in Listing~\ref{lst:submission-entry}. 

\vspace{0.5cm}
\begin{lstlisting}[caption={Submission queue entry structure.}, label={lst:submission-entry}]
struct submission_queue_entry
{
    operation_t operation_code;
    uint32_t parameter1;
    uint32_t parameter2;
    size_t buffer_index;
} typedef submission_queue_entry_t;
\end{lstlisting}

Submission and completion queues are implemented as fixed-size circular buffers, each maintaining head and tail pointers. Using fixed-size entries allows direct indexing via the head and tail pointers, simplifying access and improving efficiency. Yet, this requires all entries to share a common structure, meaning some fields may be unused for certain operations. As the size of each entry is small, however, this overhead has minimal impact on overall memory usage. 

\newpage
New entries are written at the tail, while entries are consumed from the head. The producer updates the tail pointer after adding entries, and the consumer updates the head pointer after removing them. Before adding or removing entries, the state of the queue must be checked to ensure it is not full or empty, respectively, by comparing the head and tail pointers. As a result, these pointers must be shared between the client and the server. 

As Microkit only allows memory sharing at page granularity, with a minimum mapping size of 1~KiB, each client is allocated a contiguous region of shared memory with the file server, simplifying the static system description. This region is sized to accommodate all required data structures, including queues, buffers, allocation tables, pointers, and control flags.

This memory is organised as a single structured layout, as defined in the file server API. This also allows the relative sizes of internal structures to be adjusted without affecting the overall interface. The structure is shown in Listing~\ref{lst:client-data}.

Similarly to the synchronous setup, all client memory regions are mapped into the file server’s address space at specified virtual addresses during system initialisation. As these regions are of uniform size and arranged sequentially, the server can index directly using the client’s channel identifier to access the corresponding client state. Encapsulating all client state within a single structure simplifies function interfaces, as only a pointer to this structure needs to be passed, enabling a modular implementation of the file system. 

\vspace{0.6cm}
\begin{lstlisting}[caption={Client shared data structure.}, label={lst:client-data}]
struct client {
    size_t submission_queue_head;
    size_t submission_queue_tail;
    size_t completion_queue_head;
    size_t completion_queue_tail;
    client_flags_t flags;
    submission_queue_entry_t submission_queue[MAX_QUEUE_ENTRIES];
    completion_queue_entry_t completion_queue[MAX_QUEUE_ENTRIES];
    bool submission_buffer_table[NUMBER_OF_BUFFERS_PER_CLIENT];
    bool completion_buffer_table[NUMBER_OF_BUFFERS_PER_CLIENT];
    buffer_t submission_buffers[NUMBER_OF_BUFFERS_PER_CLIENT];
    buffer_t completion_buffers[NUMBER_OF_BUFFERS_PER_CLIENT];
    uint8_t padding[CLIENT_DATA_PADDING_AMOUNT];
} typedef client_t;
\end{lstlisting}

\newpage
Once the desired number of submission entries have been added, the client sets its ready flag and signals the server using Microkit’s \texttt{microkit\_notify} function. This sends a notification along a statically defined channel without blocking the client, allowing it to continue issuing further requests or performing other work. This ready flag allows the server to efficiently poll clients when required and ensures that only clients that have explicitly indicated pending work are serviced. 

If the server is idle, receiving a notification causes its thread to be scheduled. Execution begins at the designated entry point: 

\begin{lstlisting}
void notified(microkit_channel ch)
\end{lstlisting}

The \texttt{channel} identifier is used to determine which client issued the notification, allowing the server to access the corresponding shared memory region. The server then checks that the client’s submission queue is valid and begins processing any pending entries. 

To service an operation, the server removes the next entry from the submission queue, incrementing the head, and extracts its parameters. If the operation requires a completion buffer, one is acquired using the same allocation mechanism as the client. If no buffer is available, the operation fails and an appropriate return code is generated. Otherwise, the operation is performed, and any result data is written to the allocated completion buffer. 

A completion queue entry is then constructed containing the result code, any return values, and the buffer index. This entry is added to the completion queue. The structure of this entry is shown in Listing~\ref{lst:completion-entry}.

\begin{lstlisting}[caption={Completion entry structure.}, label={lst:completion-entry}]
struct completion_queue_entry
{
    uint8_t return_code;
    uint32_t result1;
    uint32_t result2;
    size_t buffer_index;
} typedef completion_queue_entry_t;
\end{lstlisting}

Once the operation is complete, the submission buffer is released by marking its entry in the allocation table as free, as its contents have been consumed. The server then continues processing entries from the submission queue until it is empty or a predefined per-client batch limit is reached, preventing any single client from monopolising the server.

When processing is complete, the server updates the client’s flags to indicate that results are available, and that no outstanding requests remain (if processing was not terminated early), before sending a notification to wake the client if it is idle.

\newpage
As notifications are not queued, the server cannot rely on them as the sole trigger for processing requests. After servicing a client, it polls all clients by checking their ready flags. This ensures that requests are not missed if a notification arrives while the server is already processing work. The server iterates over all clients, servicing any with pending requests. 

On the client side, once notified or when explicitly checking for completion, the client removes entries from the completion queue and processes the results. After consuming any returned data, it releases the corresponding completion buffers by updating the allocation table, making them available for future operations. 

\subsubsection{API}

Much of this communication complexity is abstracted away by library functions, providing a simple and consistent interface for interacting with the file server. As in the synchronous implementation, each file system operation is exposed through a dedicated wrapper function.

For example, \texttt{send\_open\_file\_request} takes a file name, the desired opening mode and a pointer to the client’s shared state structure. It first attempts to acquire a free submission buffer, failing if none are available, and then copies the file name into that buffer. Using the acquired buffer index and the supplied parameters, it constructs a \texttt{submission\_queue\_entry\_t}, inserts it into the submission queue if space is available, and advances the submission queue tail. An appropriate result code is then returned to the client to indicate whether submission was successful.

As request submission is non-blocking, additional helper functions are required both for notifying the server and for collecting results. Two notification mechanisms are provided. The first uses \texttt{microkit\_notify} to signal the file server and then returns immediately, allowing the client to continue execution when further independent work is available. The second is intended for cases where subsequent processing depends on completion of the submitted operation or operations. In this case, \texttt{microkit\_ppcall} is used to suspend the client thread until the server has processed the request, similarly to the synchronous system, but without transferring request data. This is more efficient than busy waiting when no further useful work can be performed. Both the \texttt{notified} and \texttt{protected} server entry points then delegate to the same \texttt{service\_client} function, ensuring that request handling remains identical regardless of how servicing was triggered.

If the client does not block when notifying the server, it must later check whether results are available. This can first be done by inspecting the \texttt{complete} flag. However, if multiple operations were submitted as a batch, the flag may be set before all expected results have been completed, in which case the completion queue head and tail pointers must also be compared. As no useful work may remain while waiting for these results, dedicated wrapper functions were implemented to either busy wait on these conditions or yield execution, returning only once the requested operations have completed.

Results are then retrieved using \texttt{get\_next\_completion\_entry}. After any returned data has been consumed, the associated completion buffer may be released by the client through a dedicated wrapper function.

\section{Error Handling}

The primary sources of errors in this system arise from invalid client requests and resource limitations. These include cases such as invalid permissions, malformed paths or file descriptors, as well as exhaustion of resources such as available blocks or file descriptors. To maintain correctness, the appropriate conditions are explicitly checked before the execution of each stage of the operation.

To support this, operations are decomposed into smaller functions, each responsible for a specific task. At each potential point of failure, inputs and system state are validated, with failures resulting in the return of a defined error code from a global set. These error codes are propagated through the call chain, allowing higher-level operations to respond appropriately. Where possible, alternative actions are attempted; otherwise, the operation is safely terminated, reverting any partial changes to maintain consistency. The final error state is then communicated back to the client through the response structure. Some valid, yet undesirable, conditions such as trying to read a size larger than the buffer size, still cause the client to be notified that the operation was not fully successful.
